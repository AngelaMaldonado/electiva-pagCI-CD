Punto 1

¿Cuáles son las prácticas recomendadas por AWS para la adopción de la nube híbrida?

  • Utilización de Infraestructura Existente:

    Las organizaciones pueden aprovechar las inversiones realizadas en sus instalaciones al migrar aplicaciones a la nube, obteniendo la agilidad y escalabilidad de AWS. Durante este proceso, las herramientas como AWS Migration Hub y AWS Application Migration Service facilitan la migración sin interrumpir el funcionamiento.

  • Migración de VMware:

    Las empresas que utilizan VMware pueden migrar sus cargas de trabajo de manera fluida a AWS mediante la integración nativa de VMware Cloud on AWS, permitiendo una transición gradual sin reconfiguraciones significativas.

  • Computación Híbrida y Escalabilidad:

    AWS permite ampliar el centro de datos mediante servicios que facilitan la extensión de la infraestructura local a la nube. Esto incluye escenarios de escalabilidad, respaldo y recuperación ante desastres. AWS Outposts es ideal para aquellas empresas que necesitan ejecutar servicios de AWS en sus propias instalaciones.

  • Ampliación Geográfica y Cumplimiento:

    Las aplicaciones se pueden implementar más cerca de los usuarios finales usando AWS Local Zones o AWS Wavelength, lo que permite reducir la latencia y cumplir con los requisitos locales de residencia de datos o normativas.

  • Procesamiento Local:

    Con dispositivos como AWS Snowball y soluciones de computación periférica como AWS Wavelength, las empresas pueden procesar datos en el borde de la red antes de transferirlos a la nube, reduciendo la latencia y optimizando el rendimiento de las aplicaciones móviles y las redes 5G.

  • Admitir Aplicaciones de Baja Latencia:

    AWS permite colocar infraestructura cerca de los usuarios para cumplir con los requisitos de baja latencia, como milisegundos en aplicaciones sensibles, utilizando AWS Local Zones o AWS Outposts.

  • Servicios en las Instalaciones y la Nube:

    La integración de aplicaciones entre los entornos on-premises y en la nube se facilita con herramientas de gestión unificada, como AWS Systems Manager, que garantizan la coherencia operativa. AWS Identity and Access Management (IAM) y AWS Key Management Service (KMS) permiten mantener la seguridad y el control en ambos entornos.

  • Residencia de Datos:

    AWS ofrece opciones para cumplir con las normativas locales de almacenamiento de datos, como la residencia de datos en países específicos, utilizando los mismos servicios y herramientas que se aplican tanto en la nube como on-premises.

  • Procesamiento de Datos Localmente:

    Las arquitecturas híbridas permiten mantener ciertos datos en las instalaciones, debido a limitaciones de ancho de banda, costos o normativas, mientras que otros datos pueden ser transferidos y archivados en la nube de AWS. Soluciones como AWS Storage Gateway y AWS DataSync permiten un flujo de datos eficiente entre los entornos.

Punto 2

Investigar y responder a:

  • Es un conjunto de practicas, texting, build, deployd de aplicaciones en pequeños pasos de forma frecuente, es decir el ciclo de vida de desarrollo del software

  • CI/CD son prácticas de desarrollo de software que permiten una integración y entrega continua de código.Esto significa que los cambios en el código se integran y despliegan automáticamente en entornos de producción.

    Beneficios de CI/CD
    • Reducción de errores en producción.
    • Despliegues más rápidos y frecuentes.
    • Mejor colaboración entre equipos de desarrollo.
    Herramientas de CI/CD

    Existen varias herramientas populares que facilitan CI/CD:

    • Jenkins
    • GitHub Actions
    • Travis CI
    • CircleCI
    • GitLab CI

  • Un pipeline CI/CD es un proceso que automatiza la creación, prueba e implementación de código para el desarrollo de software. Se trata de una serie de pasos que se ejecutan en orden para distribuir una nueva versión de un sistema de software.

    El objetivo de un pipeline CI/CD es minimizar el error humano y mantener un proceso consistente para la publicación del software.

    Los pipelines CI/CD son una combinación de:

    • Código
    • Procesos, como crear y probar flujos de trabajo
    • Herramientas

  • Git:

    Git es un sistema de control de versiones distribuido que permite a múltiples personas trabajar en el mismo proyecto al mismo tiempo, realizando cambios y colaborando sin riesgo de perder información. Git rastrea todos los cambios en el código fuente a lo largo del tiempo, permitiendo a los desarrolladores retroceder a versiones anteriores si es necesario, y facilita la integración de cambios de diferentes colaboradores.
  • GitHub:
    • GitHub es una plataforma basada en la web que utiliza Git para alojar proyectos de software. Además de las funciones de Git, GitHub ofrece herramientas adicionales para la colaboración, como:
    • remotos: Puedes almacenar tus proyectos de Git en la nube, permitiendo que otros los accedan y contribuyan.
    • Colaboración: GitHub facilita la colaboración a través de "pull requests", revisiones de código y "issues" (problemas) para el seguimiento de tareas o errores.
    • Integraciones: GitHub se integra con diversas herramientas de desarrollo continuo y despliegue automatizado.
    • Proyectos públicos o privados: Puedes hacer que tus repositorios sean accesibles al público o mantenerlos privados.

GitHub Actions es una herramienta de CI/CD integrada en GitHub que permite automatizar flujos de trabajo directamente en el repositorio. Permite a los desarrolladores crear acciones que se ejecutan en respuesta a eventos, como pushes o pull requests.

Al implementar GitHub Actions, los equipos pueden:

  • Automatizar pruebas y validaciones de código.
  • Desplegar aplicaciones automáticamente en entornos de producción.
  • Personalizar flujos de trabajo mediante el uso de YAML para definir acciones y secuencias de comandos.

Esto mejora la eficiencia y reduce el riesgo de errores, ya que las acciones se pueden configurar para ejecutarse automáticamente cada vez que se produce un cambio en el código.

Conceptos clave en GitHub Actions:

1. Workflows (Flujos de trabajo): Los eventos son lo que inicia un workflow. Un evento puede ser cualquier acción que suceda en tu repositorio. Algunos ejemplos incluyen:

  • Los workflows se definen en un archivo YAML en la carpeta .github/workflows/.

2. Eventos (Triggers):Un job es una unidad de trabajo que se ejecuta en un entorno de ejecución (por ejemplo, una máquina virtual o un contenedor). Un workflow puede tener uno o varios jobs, y estos pueden ejecutarse en paralelo o secuencialmente.

  • push:Cuando se suben cambios a una rama.
  • pull_request:Los workflows se definen en un archivo YAML en la carpeta .github/workflows/.
  • schedule:Para ejecutar un workflow en horarios específicos.
  • workflow_dispatch:Permite disparar el workflow manualmente.

    • Los eventos son esenciales para definir cuándo deben ejecutarse los flujos de trabajo.

3. Jobs (Tareas): Un job es una unidad de trabajo que se ejecuta en un entorno de ejecución (por ejemplo, una máquina virtual o un contenedor). Un workflow puede tener uno o varios jobs, y estos pueden ejecutarse en paralelo o secuencialmente.

  • Cada job tiene su propio entorno y puede depender del resultado de otro.
  • Un job puede contener múltiples steps (pasos) que se ejecutan secuencialmente dentro de ese job.

4. Steps (Pasos): Los steps son las acciones individuales que se ejecutan dentro de un job. Un step puede ser una acción predefinida o un script personalizado. Los steps dentro de un job se ejecutan en orden secuencial.

  • Por ejemplo, un step podría ser instalar dependencias, otro podría ser ejecutar pruebas, y otro podría desplegar la aplicación.
  • Los steps pueden ejecutar comandos de shell o utilizar acciones reutilizables de GitHub (llamadas Actions).

5. Actions (Acciones): Las actions son unidades reutilizables de código que pueden realizar tareas específicas dentro de un step. GitHub ofrece muchas acciones preconstruidas, como por ejemplo una para configurar el entorno de Node.js o para hacer despliegues automáticos.
